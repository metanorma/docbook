# Architecture

The Metanorma DocBook gem is organized into distinct modules, each
responsible for a specific phase of document processing. This page
describes the purpose of each module and how they interact.

== Module structure

=== Docbook::Document

The entry point for parsing. `Document.from_xml` accepts an XML string,
detects the root element name (such as `book` or `article`), looks up
the corresponding element class in `ROOT_ELEMENT_MAP`, and delegates
to that class's `from_xml` method.

=== Docbook::Elements

Contains 160+ element classes, one per supported DocBook XML element.
Each class inherits from `Lutaml::Model::Serializable` and declares
typed attributes and XML mappings using the lutaml-model DSL. Classes
are autoloaded on demand from `lib/docbook/elements/`.

=== Docbook::XIncludeResolver

Resolves XInclude directives before model parsing. It iterates through
`xi:include` elements in a Nokogiri document, fetching referenced files
and replacing the include elements with the included content. Supports
both XML and text parse modes, plus `fragid` extensions for line-range
and search-based text extraction.

=== Docbook::XrefResolver

After parsing, builds an O(1) hash map of `xml:id` values to their
corresponding element objects. The resolver walks the element tree
recursively, collecting every element that has an `xml_id`. Callers
then use `title_for(linkend)` to resolve cross-reference text.

=== Docbook::Output::SinglePage

The complete pipeline for building interactive HTML readers. It takes
an XML file path and produces a self-contained HTML file with the
Vue SPA, DocbookMirror JSON data, and all assets inlined. This is the
primary output path, used by the `build` CLI command.

=== Docbook::Output::DocbookMirror

Exports parsed DocBook documents as ProseMirror-compatible JSON via the
`Mirror::Transformer`. This converts Docbook element objects into a
`{doc: {content: [nodes]}}` tree with typed nodes and marks. Used by the
`export` CLI command and embedded in SinglePage output.

=== Docbook::Models

A set of pre-computed models (`DocumentRoot`, `TocNode`,
`SectionRoot`, etc.) that represent the final JSON structure consumed
by the Vue SPA. They include additional metadata like reading position
and generator version.

=== Docbook::CLI

A Thor-based command-line interface that provides the `build`,
`export`, `validate`, and `format` commands (plus `--demo` on `build`
for the bundled sample). It wires together the parsing, resolution, and
output modules based on user options.

== Element model system

Every DocBook element is represented by a Ruby class that inherits from
`Lutaml::Model::Serializable`. This provides:

* **Typed attributes** via the `attribute` DSL method
* **Mixed content** support via the `mixed_content` declaration
* **XML serialization** via `from_xml` / `to_xml`
* **JSON serialization** via `to_json`

The `each_mixed_content` method yields child nodes (both `String` and
element objects) in document order, which is essential for correct
inline content rendering.

== Frontend architecture

The Vue 3 SPA in `frontend/` follows a composable-based architecture
with clear separation of concerns.

=== Composables

Each composable (`frontend/src/composables/`) encapsulates a single
feature with its own state, lifecycle, and cleanup. Composables use
Vue's `provide`/`inject` pattern to share state down the component tree
without prop drilling.

[cols="2,4"]
|===
| Composable | Responsibility

| `useLazySections` | IntersectionObserver-based deferred section rendering
| `usePaginatedMode` | Viewport-sized page calculation and navigation
| `useReadingRuler` | Mouse-following horizontal ruler
| `useBookmarks` | Bookmark CRUD with localStorage persistence
| `useTts` | Web Speech API text-to-speech with chunked playback
| `useReadingStats` | Reading progress tracking and word count metrics
| `useCloudSync` | Extensible sync via `SyncBackend` interface
| `useEbookStore` | Theme, font, and reader settings management
|===

=== Rendering pipeline

. `documentStore.loadFromWindow()` reads embedded JSON into `shallowRef`
. `useLazySections` creates an IntersectionObserver with 2000px root margin
. `MirrorRenderer` recursively walks the DocbookMirror node tree
. Non-visible sections render as lightweight placeholders
. Sections become visible as they approach the viewport

=== Open/Closed Principle

The frontend follows OCP in several places:

* **SyncBackend interface** -- new cloud backends (WebDAV, Google Drive)
  implement the `SyncBackend` interface without modifying `useCloudSync`
* **MirrorRenderer node dispatch** -- adding a new node type requires
  only adding a `v-else-if` block in the template; no existing code changes
* **Transformer container mapping** -- new element types map to handler
  methods via a dispatch hash; adding a type doesn't modify existing handlers
